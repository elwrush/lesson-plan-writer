# Plan: Slideshow-to-PPTX skill — optimized pipeline

## Objective

Convert a reveal.js HTML slideshow (`index.html` + `assets/`) into a full-visual-fidelity PowerPoint `.pptx` where every slide is a high-resolution rasterized image. The result looks identical to the browser-rendered presentation — backgrounds, icons, fonts, colors, and layout all preserved.

## Optimized Pipeline (2 phases, no redundant rendering)

### Phase 1: Decktape → PDF
Decktape uses Chrome headless (Puppeteer) to open the reveal.js HTML and render each slide as a PDF page. This is the **single rendering pass** — no `--screenshots` flag.

```powershell
decktape reveal "file:///.../slides/index.html" "slides.pdf" --load-pause 2000 --chrome-arg=--allow-file-access-from-files
```

### Phase 2: PDF → PPTX via PyMuPDF + python-pptx
A single script extracts each PDF page as a 1920×1080 PNG (using PyMuPDF), then builds a PPTX with each PNG as a full-slide picture (using python-pptx). This replaces the old 2-step (screenshots + build).

```powershell
python scripts/pdf_to_pptx.py "path/to/slides.pdf" "path/to/output.pptx"
```

## Why This is Faster

| Old pipeline (3 phases) | New pipeline (2 phases) | Saving |
|---|---|---|
| Decktape renders + captures PDF | Decktape renders + captures PDF | Same |
| Decktape ALSO captures `--screenshots` (duplicate render per slide) | **Eliminated** | ~15-20% of decktape time |
| Build script reads PNGs from disk | PyMuPDF extracts pages from PDF (in-memory) | No intermediate file I/O |
| Path bug: `--screenshots-directory` needs Push-Location workaround | **Eliminated** | No workaround needed |
