# Skill: Publish to GitHub Pages

## Purpose
Publish a mkslides-generated `site/` directory to GitHub Pages so the presentation is accessible via a public URL.

## Prerequisites
- `gh` CLI must be installed and authenticated (`gh auth status`)
- Repository must be on GitHub
- mkslides build must already be complete (`site/` directory exists)

## Workflow

### Step 1: Verify inputs
- Check that `site/` directory exists with `index.html`
- Detect GitHub remote URL: `git remote get-url origin`
- Extract username and repo from remote URL

### Step 2: Publish to gh-pages branch

Use the `gh` CLI and git to push the site:

```bash
# Create gh-pages orphan branch if it doesn't exist
git checkout --orphan gh-pages

# Remove existing tracked files (keep .git)
git rm -rf .

# Copy site contents to root
Copy-Item -Recurse -Force "output/{subfolder}/site/*" .

# Commit and push
git add -A
git commit -m "Deploy slides: {topic} ({date})"
git push origin gh-pages --force

# Return to original branch
git checkout {original_branch}
```

If `gh-pages` branch already exists:
```bash
git checkout gh-pages
# Copy site contents over
Copy-Item -Recurse -Force "output/{subfolder}/site/*" .
git add -A
git commit -m "Update slides: {topic} ({date})"
git push origin gh-pages
git checkout {original_branch}
```

### Step 3: Enable GitHub Pages (if needed)
```bash
gh api repos/{owner}/{repo}/pages --method POST -f "source[branch]=gh-pages" -f "source[path]=/"
```
This is idempotent — safe to run even if already enabled.

### Step 4: Return the URL
- URL format: `https://{owner}.github.io/{repo}/`
- If publishing to a subfolder (multiple presentations): `https://{owner}.github.io/{repo}/{presentation}/`

## Publishing Subfolder (Multiple Presentations)
To publish each presentation to its own URL:
```bash
# In gh-pages branch, create subfolder
mkdir -p {presentation}
Copy-Item -Recurse -Force "output/{subfolder}/site/*" {presentation}/
git add -A
git commit -m "Add slides: {presentation}"
git push origin gh-pages
```
URL: `https://{owner}.github.io/{repo}/{presentation}/`

## Constraints
- Must NOT destroy existing gh-pages content
- Must preserve index.html for other presentations
- Must work with already-authenticated gh CLI
- Must return to the original branch after push
- Force-push to gh-pages is acceptable (it's a deployment artifact, not source)

## Error Handling
- If gh CLI not installed: instruct user to install (`winget install GitHub.cli` or `scoop install gh`)
- If gh not authenticated: instruct user to run `gh auth login`
- If push fails (no permission): report the error and suggest checking repo permissions
- If GitHub Pages not enabled: suggest enabling in repo settings or via gh CLI
